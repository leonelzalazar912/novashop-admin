---

## 1. Visión del producto

NovaShop será una plataforma comercial adaptable a pequeños y medianos negocios de cualquier rubro.

No es solamente una tienda online. El producto final combina:

- Ecommerce público.
- Panel administrativo.
- Gestión de productos e inventario.
- Compras y proveedores.
- Pedidos, ventas, clientes y caja.
- Reportes.
- Usuarios, roles y permisos.
- Configuración personalizada para cada comercio.

Debe poder utilizarse para videojuegos, ropa, celulares, perfumería, librería, repuestos, electrónica, hogar y otros rubros sin modificar el código fuente.

El objetivo final es comercializar NovaShop como una base reutilizable y configurable para diferentes negocios.

8. Decisiones tomadas
Decisión 1 — Supabase será la infraestructura definitiva
La aplicación dejará de usar localStorage como persistencia principal.

localStorage podrá utilizarse únicamente para:

Preferencias no sensibles.

Caché.

Estado temporal del carrito, cuando corresponda.

Configuración visual local no crítica.

Decisión 2 — No crear más módulos antes de la infraestructura
Se detuvo el inicio de Ventas y de otros módulos.

Antes de continuar se debe:

Auditar los modelos actuales.

Diseñar la base completa.

Crear autenticación y seguridad.

Crear una capa de servicios.

Migrar los módulos existentes.

Decisión 3 — NovaShop será multiempresa
Los datos comerciales deberán pertenecer a un comercio.

Las tablas relevantes deberán incluir una relación equivalente a:

store_id
Los usuarios solo podrán acceder a comercios donde tengan autorización.

Decisión 4 — Usar UUID
Los IDs definitivos no deben generarse con:

Date.now()
Supabase/PostgreSQL generará UUID.

Decisión 5 — No guardar relaciones como texto
No usar como relación definitiva:

category: string
brand: string
supplier: string
Usar identificadores:

category_id
brand_id
supplier_id
Decisión 6 — Separar interfaz y persistencia
Los componentes React no deben acceder directamente a Supabase.

Flujo recomendado:

Componente
→ hook
→ servicio o repositorio
→ Supabase
Decisión 7 — Diseñar antes de ejecutar SQL
No crear tablas manualmente una por una.

Primero revisar todos los modelos necesarios.

Después preparar y revisar un diseño consolidado.

Finalmente ejecutar una migración SQL inicial.

9. Error estratégico detectado
Se construyeron varios módulos completos utilizando localStorage, aunque desde el inicio estaba prevista una base de datos real.

Esto generará retrabajo principalmente en:

Hooks.

Persistencia.

IDs.

Relaciones.

Manejo asíncrono.

Estados de carga.

Manejo de errores.

Seguridad.

Movimientos de stock.

No se pierde la mayor parte del trabajo visual:

Formularios.

Tablas.

Modales.

CSS.

Exportación CSV.

Impresión.

Filtros.

Ordenamiento.

Reglas funcionales.

Regla resultante:

No desarrollar módulos nuevos sobre una infraestructura que sabemos que será reemplazada.

15. Reglas de trabajo
Avanzar de a un paso cuando la tarea sea nueva para la persona que programa.

No entregar muchos cambios simultáneos sin necesidad.

Cuando se modifica un archivo de forma importante, entregar el código consolidado.

No afirmar que un archivo queda cerrado si existe una dependencia futura evidente.

Revisar el impacto de cada cambio en todo el producto.

Priorizar infraestructura definitiva antes que módulos nuevos.

Evitar duplicar hooks o fuentes de datos.

No guardar relaciones comerciales como texto.

No acceder a Supabase desde componentes.

No ejecutar SQL sin revisar antes el modelo.

No exponer claves privadas.

No usar la clave service_role en el frontend.

Mantener el proyecto genérico para cualquier rubro.

Actualizar este archivo al terminar cada sesión importante.

Registrar siempre el próximo paso exacto.
